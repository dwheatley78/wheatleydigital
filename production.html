Developing a Data Platform

While data analysis can provide rich insights and value for an organisation only 10% of developed machine learning models make it into production to deliver real value. The two main problems preventing wider adoption of analytical solutions are:

  • Unreliable source data - If developed models are built on uncontrolled data sources, e.g. local spreadsheets and databases errors, inconsistencies and access issues will occur. It is essential to provide robust and validated data repositories to secure the capability longer term.
  • Accessibility – If the developed solutions are not accessible and intuitive to prospective users there true value will never be realised. Furthermore the look and feel of the applications must match the company values, especially when providing services directly to your customers.

Developing an intuitive user interface (UI) for both the consumption and collection of data is essential. Validating data entry, managing access and permissions and providing coherence with the wider corporate data warehouse strategy are essential. As the business becomes more dependent on the applications the code and models developed need to be reliable, maintainable and reproducible.

All this can be achieved through the adoption of good software development practices and appropriate tool selection. In this section we outline

  • DevOps - software industry best practice way of working
  • WebApps – defining the architecture of a web app
  • Tech Stack – Selecting the tools to develop an application
  • Hosting – Make an application accessible

DevOps

DevOps has become the software industry best practice way of working. Combining cultural philosophies, practices, and tools to develop and evolve applications faster and more reliably.

At its heart the objective is to improve communication and feedback, automate processes where possible and identify issues early. By breaking a large problem into smaller units, constantly validating the output with users and performing rigorous testing gives the best possible chance of success.

The lifecycle of a production solution can be represented as a figure of eight as the requirements continue to evolve over time. Here we will introduce a few key concepts to help business leaders to understand however for a more comprehensive account of DevOps practices refer to the DevOps Handbook.

×
  • Minimum Viable Product (MVP) - When developing a new application we always seek to produce something with enough features to get internal business users using the system or attract early-adopter customers with whom we can validate the concepts early. Once created we can then iteratively develop the capability learning from the feedback received.
  • Agile – When operating with a high degree of uncertainty traditional project management techniques don’t work. Agile is an approach to project management in which small 2-3 week sprints of activity are planned and aim to deliver some demonstrable output, which can be tested with the user.
  • Continuous integration – is a software development practice where developers regularly merge their code changes into a central repository, after which automated tests validate the code quality. A robust approach to ensure good test coverage enables bugs to be found quicker, improve software quality, and reduce the time it takes to validate and release new software updates.
  • Application Programming Interface (API)) –In computer programming a guiding design principle is to introduce Separation of concerns (SoC). Breaking the programme into distinct services that communicates with other services through a well-defined interface such as an API. This approach allows areas of the code to be developed in parallel and minimises the risk of unintended consequences so long as the interfaces between services are maintained.
  • MLOPS– Extends the concept of DevOps to machine learning models where integration not only covers testing and validating code but is now extended to test and validate the data in the system including the retraining and serving of models as required.

WebApps

When building applications the main objective is to make the functionality and data contained within it available to a range of users, both internal and external to the organisation. Web apps provide secure access to a range of users over the Internet through their web browser. This ability to avoid the need to install specialist software on peoples devices, allowing all updates to be deployed remotely and centralising the data store make WebApps an ideal solution for most applications.

Streamlit is a really useful Python library that enables simple WebApps to be produced without any front end web development, making them an ideal means of sharing analytical models across an internal team.

For more complex solutions a more traditional approach to building applications will be requried, splitting the functionality across server and client side processing. For the majoirty of applications we favour a Model View Controller (MVC) architecture, which balances the benefits of inrtoducing a separation of concerns between the frontend and backend components while avoiding some of the complexity utilising microservices can cause. Following the user interface, which is controlled using a combination of HTML, CSS and Javascript runs in the browser, whereas the business logic, authentication and interaction with the databases occurs on the server side. In the following section we explore our prefered technology stack when building tehse applications.

  • Model - The backend that contains all the data logic
  • View – The frontend or graphical user interface (GUI)
  • Controller – The brains of the application that controls how data is displayed.

Tech Stack

The options when selecting the languages, libraries and frameworks with which to develop an application can be overwhelming. While specific applications may necessarily drive towards different solutions it is useful to default stack that can guarantee affordability, supportability, reliability, and security.

In the context of supporting data driven digital transformation the ability to incorporate machine learning models and analysis (detailed in the ANALYSE section of site) is paramount. We therefore look to use a Python framework to write our backend processes, maximising the ability to reuse code and minimising the breadth of knowledge required to manage these developed solutions.

Another key consideration is to select popular open-source frameworks for which there is a rich source of training materials and support while avoiding costly licence fees. For these reasons, our default stack uses Django, React and PostgreSQL, which collectively provide a highly capable and supportable solution.

A brief outline of each of these is provided below.

  • PostgreSQL – provide a highly capable database to store the required data. It supports a wide variety of data types, provides integrity controls to prevent invalid or orphan records and powerful indexing methods to optimize performance. Advanced features such as full-text search speed up finding information and the ability to store non-relational data such as JSON and XML formats actually turns PostgreSQL into NoSQL database providing the flexibility to meet a the evolving requirements of your applications.
  • Django - is a Python backend (server-side framework) able to integrate easily with existing analytical and machine learning models. Django provides a host of built in advanced functionality such as admin panels, file upload support, authentication, comment boxes, contact forms, app management and a versatile toolkit for constructing web APIs to significantly reduce the time and effort in building robust web applications.
  • React – is a JavaScript frontend (client-side framework used to build single page User Interfaces (UI). getting and setting data via requests to the Django backend. As with Django, React comes with multiple component libraries, add-on packages and full template offerings to save developers a lot of time capable of producing engaging and intuitive user interfaces.

Data Architecture

3 terms are often used when talking about data storage:

Data Lake

Contains raw data, of various formats, in its native form

Data Warehouse

A Data warehouse is a central repository that merges information from different sources and consolidates it through the extract, transform and load process, also known as ETL process, into a comprehensive database used for analytical and business techniques. At a very high level, the ETL process extracts the data from multiple sources, transforms it into a cleaned format to be used for business processes, and finally loads that data into the data repository. A data warehouse stores current data and the historical information that has been cleansed, conform categorized. When data gets loaded into the data warehouse, it is modelled and structured, ready for a specific purpose.

Data Marts

Data Marts can be created in 3 different ways all of which aim to provide the requried data to the end-users and make it accessible to the wider organisation. These data marts accelerate the business process and offer a cost and time-efficient way to take the data-driven decision.

  • Dependent data marts - built by drawing data from the existing central warehouse generally used for isolated security and solo performance.
  • Independent data marts - Created by drawing from operational or external data sources normally used to capture business or customer data that would otherwise reside with individuals and be inaccessible to the wider organisation.
  • Hybrid data mart - combines input from a data warehouse, operational systems and external systems.


Hosting

There are fundamentally two types of hosting arrangement available.

Statically hosting- enables web pages serve up the same page to HTML, CSS and JavaScript to all users who enter the site, in reality this is nothing more than a file store. As no processing is required on the server these pages are generally faster and can be hosted at minimal, or no, cost using a variety of service providers including our favourite GitHub Pages.

Dynamic hosting -enables the processing of content and storage of data on the server. Dynamic hosting can be provided on any corporate network or by a cloud service provider however these services can become complex to manage. If you have a team already set, up to configure and manage servers these options provide the greatest flexibility however they can be difficult to manage and control cost if you don’t have the prerequisite skills.

An alternative solution is to opt for a PaaS (platform as a service) offering a more managed, low configuration services, making it incredibly easy to get up and running and provide total upfront transparency of the costs. Our favoured solution is to use Heroku, which comes with the added benefit of providing a “Free Tier” which, while there are significant constraints in terms of speed and storage capacity, provides an ideal sandpit environment in which to demonstrate capability without commitment.

Aircraft Prognostics


Autonomous aircraft, requiring no human interventions during operation, is a rapidly evolving sector. The global autonomous aircraft market was estimated to be USD 4.56 billion in 2019 and is projected to reach USD 16.23 billion by 2027.

With such competition and growth potential manufacturers are looking towards enhanced service strategies as a means of delivering a competative advantage

In responce to this opportunity we were commissioned to perform a comprehensive review intelligent health management solutions and develop a prototype solution utilising advanced analytics to demonstrate an approach to deliver adaptive vehicle health management, providing greater process control, reduced operating costs and enhanced availability.



The approach taken involved decomposing this top-level objective into three targeted value propositions.

  • Accurately predicting the remaining useful life of aircraft components based on the provided sensor data
  • Providing dynamic fault diagnostics advice to the maintainer based on the observations
  • Simulating the degradation of the aircraft and the potential maintenance decisions to determine the optimum responce based on defined objective funcitons and within defined operational constraints.


Aviation will always mandate stringent safety requirements, however the ability to reduce uncertainty regarding the health of components presents an enormous opportunity to reduce cost and drive operational efficiencies.

The novel approach developd here, normalising the aircraft sensors data based on flight condition, to train a Long-Term-ShortTerm-Memory (LSTM) model demonstrated the ability to predicting the degradation state of these complex systems, with supporting confidence intervals, and demonstrate how a move to a condition-based maintenance strategy could be realised.

The use of Bayesian Belief Networks (BBNs) provided the requried adaptive diagnostic advice to the maintainer in an intuitive and visual way. This approach had the additional benefit of enabling any divergence from the designed system reliability

Performing a monte-carlo analysis on the simulation combined with a dual annealing optimisation function enabled:

  • An optimised stocking policy to be proposed, based on both financial and aircraft availability constraints.
  • Opportunity maintenance to be identified (proactively replacing an item early) to reduce cost and better utilise limited manpower and hangar capacity
