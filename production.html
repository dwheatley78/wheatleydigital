Developing a Data Platform

Data analysis can provide rich insights for an organisation; however, the real value is only realised once these tools are made available to a wider user base. It is estimated that only around 10% of machine learning models developed actually make it into production, and hence the real value is never realised. The two main problems preventing wider adoption of analytical solutions are:

  • Unreliable source data - While manual collection and bespoke data preparation can support isolated analysis a more robust data pipeline is required to allow an application to scale. It is therefore essential to develop robust and validated data repositories to secure the data capture and preparation process for the longer term success of a solution.
  • Accessibility – If the developed solutions are not accessible and intuitive to prospective users their true value will never be realised. Furthermore, when providing services directly to your customers, it is essential that the look and feel of the application match the company values you wish to portray.

Developing an intuitive user interface (UI) for both the consumption and collection of data is essential. Validating data entry, managing access and permissions and providing coherence with the wider corporate data warehouse strategy is essential. As the business becomes more dependent on the applications the code and models developed need to be reliable, maintainable and reproducible.

On this page we discuss some of the tools and software practices that can help ensure effective solutions are developed.

  • Designing the solution – Defining the architecture of a web app

Effective Development

DevOps has become the software industry best practice way of working, combining cultural philosophies, practices, and tools to develop and evolve applications faster and more reliably.

At its heart DevOps is about creating flow between the various process stages, represented by a figure of eight shown. Through improving communication and feedback, identify issues early and automating processes where possible gives the team the best possible chance of success.

We outline a few key concepts that should be understood, however for a more comprehensive account of DevOps practices refer to The DevOps Handbook.

  • Agile – When operating with a high degree of uncertainty traditional project management techniques don’t work. Agile, as defined in more detail in the book Scrum The Art of Doing Twice The Work In Half the Time, is an approach to project management in which small 2-3 week sprints of activity are planned, each of which deliver demonstrable capability, which can be tested with the user.
  • Minimum Viable Product (MVP) - When developing a new application we need produce something with just enough features to encourage users to adopt the solution. These users then support its iterative evolution constantility testing enhancements and learning from the feedback received.
  • Version control systems - Systems such as Git provide a means of storing, sharing and tracking changes to source code in a special kind of database. This repository is an invaluable source of that needs to be protected from both catastrophic events and the casual degradation of human error and unintended consequences.
  • Code Formatting While the software your code communicates with doesn’t care how the code looks, other developers who may have to work on it in the future certainly do. When everyone is writing code that looks the same, your brain can relax a bit as the understanding comes faster. Using opinionated formatting tools, like Black can enforce rules automatically to simplify the process of writing well structured code.
  • Testing - is fundamental to ensure good quality code is being written and allows changes to be quickly validated. The main types of testing used are:
    • Unit Testing - refers to taking a component of a program and testing it in isolation by writing functionally independent components unit tests can be easily automated to validate changes, test dependencies and handling of edge cases.
    • Functional Testing - checks the compliance of the system overall against a defined set of criteria. In other words, does the software as a whole do what it’s supposed to do?
    • Regression Testing - running previously written tests whenever a new change is introduced to the code allowing you to remain confident that your code continues to function as expected as it grows in complexity and capability.
  • Continuous integration – is a software development practice where developers regularly merge their code changes into a central repository, after which automated tests validate the code quality. A robust approach to ensure good test coverage enables bugs to be found quicker, improve software quality, and reduce the time it takes to validate and release new software updates.
  • MLOPS– Extends the concept of DevOps to machine learning models where integration not only covers testing and validating code but is now extended to test and validate the data in the system including the retraining and serving of models as required.

Structuring your data

The objective of a good data architecture is to ensure that all the information requires running your business is accurate, consistent and accessible. The first step in this process is often ensure that key information is stored centrally instead of distributed across a variety of personal computers and drives, known as a Data Lake. By creating this Data Lake, storing raw data, of various formats, in its native form in one common location functions from around the business can begin to develop the various analytical models discussed in the analyse page.

Having created a Data Lake, it will invariably become apparent that different versions of the same data exist, that they are contradict each other and that that significant work is still required to prepare the data into a format suitable for analysis.

A Data Warehouse merges information from these different sources and consolidates them through an extract, transform and load process, also known as ETL process, into a comprehensive database. At a very high level, the ETL process extracts the data from multiple sources, transforms it into a cleaned format, which can then be used for business processes, and finally that data into a structure data repository. When data is loaded into the Data Warehouse, it is modelled and structured, ready for a specific purpose.

The problem arises, as new unstructured data needs to be ingested from the Data Lake into the Data Warehouse, which can become impossible to control. The solution is to build dedicated applications that collect and validate the data at the point of entry and store the information in Data Marts.

As discussed in the production page these applications are aligned to core business processes and customer services to drive efficiency and effectiveness while the Data Marts themselves effectively combine to form the Data Warehouse. Depending on the data requirements of the application the data marts can be considered as either:

  • Independent - Created by drawing from operational or external data sources normally used to capture business or customer data that would otherwise reside with individuals and be inaccessible to the wider organisation.
  • Hybrid - combines input from a Data Warehouse, operational systems and external systems
  • Dependent - built by drawing data from the existing central warehouse generally used for isolated security and solo performance.

Ultimately, the data being collected needs to be stored in a database; however, there are a variety of database types to choose from, each with their own strengths and limitations.

  • Relational database are the go to solution, providing a strict structure / schema, ensuring the data contains a predefined list of components ensuring that the data held is predictable, easily assessable and significantly reducing the chance of errors.
  • Non-relational database organization are more forgiving in their structure. Instead of tables with columns and rows, they have collections of different categories allowing records of varying shape to be stored. In recent years traditional SQL databases such as Postgres have adapted to store storage of JSON and Array datatypes enabling them to provide a Hybrid database to deliver the best of both.
  • Graph databases are a relatively new form of database that capture both the entities and relationships between different data points. These databases rose to prominence with the growth of network analytics, as used extensively by Facebook and Google. Neo4J offers an effective Graph database solution which can contribute to your wider Data Warehouse capability.

Designing a solution

When building applications, the main objective is to make the functionality and data contained within it available to a range of users, both internal and external to the organisation. Web Apps provide secure access to a range of users over the Internet through their web browser. This ability to avoid the need to install specialist software on peoples' devices, allowing all updates to be deployed remotely and centralising the data store make Web Apps an ideal solution for most applications.

Streamlit is a really useful Python library that enables simple Web Apps to be produced without any front end web development, making them an ideal means of sharing analytical models across an internal team.

For more complex solutions a more traditional approach to building applications will be required, splitting the functionality across server and client-side processing. For the majority of applications, we favour a Model View Controller (MVC) architecture, which balances the benefits of introducing a separation of concerns between the frontend and backend components while avoiding some of the complexity utilising microservices can cause.

The use of Application Programming Interfaces (API's) allows a programme to be broken into distinct services that communicates with other services through a well-defined interface such as an API. This approach allows areas of the code to be developed in parallel and minimises the risk of unintended consequences so long as the interfaces between services are maintained.

Following the user interface, which is controlled using a combination of HTML, CSS and JavaScript runs in the browser, whereas the business logic, authentication and interaction with the databases occurs on the server side. In the following section we explore our preferred technology stack when building these applications.

  • Model - The backend that contains all the data logic
  • View – The frontend or graphical user interface (GUI)
  • Controller – The brains of the application that controls how data is displayed.

Selecting the tools

The options when selecting the languages, libraries and frameworks with which to develop an application can be overwhelming. While specific applications may necessarily drive towards different solutions it is useful to default stack that can guarantee affordability, supportability, reliability, and security.

In the context of supporting data driven digital transformation the ability to incorporate machine learning models and analysis (detailed in the ANALYSE section of site) is paramount. We therefore look to use a Python framework to write our backend processes, maximising the ability to reuse code and minimising the breadth of knowledge required to manage these developed solutions.

Another key consideration is to select popular open-source frameworks for which there is a rich source of training materials and support while avoiding costly licence fees. For these reasons, our default stack uses Django, React and PostgreSQL, which collectively provide a highly capable and supportable solution.

A brief outline of each of these is provided below.

  • PostgreSQL – provide a highly capable database to store the required data. It supports a wide variety of data types, provides integrity controls to prevent invalid or orphan records and powerful indexing methods to optimize performance. Advanced features such as full-text search speed up finding information and the ability to store non-relational data such as JSON and XML formats actually turns PostgreSQL into NoSQL database providing the flexibility to meet the evolving requirements of your applications.
  • Django - is a Python backend (server-side framework) able to integrate easily with existing analytical and machine learning models. Django provides a host of built-in advanced functionality such as admin panels, file upload support, authentication, comment boxes, contact forms, app management and a versatile toolkit for constructing web APIs to significantly reduce the time and effort in building robust web applications.
  • React – is a JavaScript frontend (client-side) framework used to build single page User Interfaces (UI). Getting and setting data via requests to the Django backend. As with Django, React comes with multiple component libraries, add-on packages and full template offerings to save developers a lot of time capable of producing engaging and intuitive user interfaces.

Choosing a Hosting Environment

There are fundamentally two types of hosting arrangement available.

Statically hosting- enables web pages serve up the same page to HTML, CSS and JavaScript to all users who enter the site, in reality this is nothing more than a file store. As no processing is required on the server these pages are generally faster and can be hosted at minimal, or no, cost using a variety of service providers including our favourite GitHub Pages.

Dynamic hosting -enables the processing of content and storage of data on the server. Dynamic hosting can be provided on any corporate network or by a cloud service provider however these services can become complex to manage. If you have a team already set, up to configure and manage servers these options provide the greatest flexibility however they can be difficult to manage and control cost if you don’t have the prerequisite skills.

An alternative solution is to opt for a PaaS (platform as a service) offering a more managed, low configuration services, making it incredibly easy to get up and running and provide total upfront transparency of the costs. Our favoured solution is to use Heroku, which comes with the added benefit of providing a “Free Tier” which, while there are significant constraints in terms of speed and storage capacity, provides an ideal sandpit environment in which to demonstrate capability without commitment.
